1. Introduction

Planty is a Web platform for online herbariums. A user can create plant albums and look at other public thematic albums.

Moreover, users can look up plants using a search platform with multiple filters: plant name, collection, collector, location and family.

Users that create an account will also receive recommendations of albums based on the plants they added to their own albums, therefore discovering more of what they like and are interested in.

2. Application Flow

The flow of Planty starts with the user entering the web application.

The views are the HTML files that we created, which keep the static data and are used as a connector between the server and our controllers.

The controllers handle the bussiness logic of the application. This consists of registering and logging users, searching for a plant, managing albums and, last but not least, generating recommendations.

The data used is provided by the Data Access Layer, which has access to a database and is composed of Models which represent the objects found in the tabels from the database.

3. Database

In this application, we make use of a database in order to store all the information so that the users can keep track of their albums.

The user table provides storage for all existing users on the platform. The fields stored are: full name (for a better customer experience), email (there can't be two accounts with the same email) and password.

The plant table is used to store all the plants along with their attributes. While researching other previously created applications similar to ours, we discovered that most online herbariums use quite the same fields in order to identify their plants.
Inspired by websites like Kew, Royal Botanical Gardens and Virtual Herbaria , we chose to identify plants by the following fields:

  • name - the scientific name in Latin, because it is universal
  • family - an important way of categorizing plants
  • collection - a collection helps users track the plants they are interested in better
  • collector - the collector is the person who uploaded the plant to the online herbarium
  • location - where the plant was discovered
  • image - is 64-bytes encoded and plays a huge visual role for the entire application, helps identify plants easier. All images were downloaded from Kew, Royal Botanical Gardens .

The album table stores all the albums created by users. Each album has a description where the user can provide details about it if needed.

The plant_album table associates a plant with the albums that contain that plant, so that we can easily get all the plants for an album.

4. MVC Model

Model-View-Controller (MVC) is an arhitectural pattern that separates an application into three main components: the model, the view and the controller. Each of these components has a specific role and contribution to the development of the application.

The MVC paradigm is easy to understand and to follow, and it brings to the table a couple advantages:

  • code decoupling - the code has fewer dependencies and therefore it is easy to manipulate
  • modularity - the code is easier to organize
  • modification not affecting the entire model
  • the project can be easily scaled and extended with new functionalities as the changes will not affect the current code

4.1 Model

The model in our application wil consist of the database entities created. This component is responsible for the data-related logic that the application works with, and transfers information to the Controller component.

4.2 View

This component will be represented by html files for every page in the website. It is basically the user interface logic that the user interacts with.

4.3 Controller

The controllers of the application are the connection between views and models. They process the incoming requests from the views and handle the data received from the models.

7. Technologies

7.1 PHP

The backend will be implemented in PHP, as it is accessible for us because of the documentation online and the courses for Web Technologies. Moreover, PHP works very well with database

7.2 MariaDB

Fork of MySQL, MariaDB was an easy choice as a database because it interacts with PHP very well. We used HeidiSQL as an administration tool for our database.

7.3 HMTL, CSS, Javascript

Using HTML and CSS gave us a full control of our user interface, which became easily customizable. Javascript was used for functions that interact with libraries and for other interactive actions on our website, like the animation for the side menu.

7.4 Version control - Git

We chose to upload versions of our code on Github because of the possibility to view changes and different versions of the project. This gave us an accurate view of the progress of development for Planty.

7.5 Google Charts

In order to make our website more interactive and pleasing, as well as enhance the user's experience, we used pie charts to display data for certain statistics.

7.5 JSPDF - Javascript PDF

For the export functionality, we chose to use a Javascript Library that creates PDF files using given data. The files are customizable and the possiblity to add tables helped us.
